# シンペイ開発指針

## プロジェクトの目的

このリポジトリは、ボードゲーム「シンペイ」をブラウザ上で遊べる形に実装するためのプロジェクトである。

同時に、TypeScript / JavaScript / React / Amplify を実践的に学ぶための学習用リポジトリでもある。完成品を一気に作ることよりも、小さい単位で実装し、動かし、理解した内容を残しながら進めることを重視する。

## 作りたいもの

最終的には、ブラウザ上で動くシンペイを作る。

目指す体験は次の通り。

- ユーザーがブラウザでゲーム画面を開く
- ルールに沿ってシンペイを遊べる
- 将来的には別ユーザーとオンライン対戦できる
- オンライン対戦では、ユーザーの片方がホストに近い役割を持つ
- ログインには passkey を使う
- 認証とゲーム本体は分離して実装する

ただし、最初からオンライン対戦やログインを必須にしない。まずはゲーム本体を小さく完成させ、その後に通信や認証を足していく。

## 現在のリポジトリ状態

現時点の構成は次の通り。

- フロントエンドは Vite + React
- Amplify Gen 2 の構成がある
- Cognito / passkey custom auth に向けた実験的な実装がある
- `docs/passkey-usernameless-hands-on.md` に passkey 認証の学習メモがある
- `docs/simpei rule.md` にシンペイの実装用ルールがある
- `NotesPage` は Amplify の data / storage を試すためのサンプルに近い
- シンペイのゲーム本体は、ルール文書に従ってローカル対戦から実装する

この状態を前提に、当面は「学習用のサンプルを維持しつつ、ゲーム本体を別の関心として育てる」方針にする。

## 設計方針

ゲーム本体とログイン機能は分離する。

ゲーム本体は、できるだけ認証や Amplify に依存しない形で実装する。盤面、駒、手番、合法手、勝敗判定などは、まず純粋なゲームロジックとして扱う。

ログイン機能は、ユーザー識別や将来のオンライン対戦、保存機能が必要になった段階で接続する。ログインできないとゲーム本体が動かない、という状態にはしない。

初期実装では、次の境界を意識する。

- ゲームロジック: 盤面状態、手番、合法手、勝敗判定
- UI: 盤面表示、駒の選択、操作、状態表示
- 通信: 将来の P2P 対戦用
- 認証: passkey / Cognito custom auth 用
- 永続化: 将来のユーザー設定や対戦履歴用

## 機能スコープ

### 初期スコープ

まずは同一ブラウザ内で2人が交互に操作するローカル対戦を作る。

初期スコープに含めるもの。

- シンペイの盤面表示
- 駒の配置
- 手番管理
- 合法手判定
- 勝敗判定
- リセット
- 最小限のエラー表示

初期スコープに含めないもの。

- ログイン必須化
- オンライン対戦
- P2P 接続
- 対戦履歴保存
- ランキング
- AI 対戦

### 将来スコープ

ローカル対戦が安定してから、次の順で拡張する。

- ゲーム状態をシリアライズできる形に整理する
- オンライン対戦用に、操作イベントを送受信できる形にする
- WebRTC を使った P2P 対戦を検討する
- 接続開始に必要な最小限のシグナリング方法を検討する
- passkey ログインと Cognito custom auth を接続する
- ユーザー情報や対戦履歴の保存を検討する

## 認証方針

ログインは passkey を使う方針とする。

AWS 側では Cognito custom authentication を使う想定で進める。Cognito は内部的に username を必要とするため、ユーザーに username / email を入力させない場合でも、passkey の `userHandle` から Cognito username を解決する設計が必要になる。

認証の実装はゲーム本体とは別に進める。認証が未完成でも、ローカル対戦のゲーム本体は開発・動作確認できる状態を保つ。

passkey 実装の学習と詳細手順は、既存の `docs/passkey-usernameless-hands-on.md` に集約する。この文書では、プロジェクト全体における認証の位置づけだけを扱う。

## P2P / 通信方針

最終的には、ユーザー側にホストに近い役割を持たせる P2P 対戦を目指す。

ただし、初期段階では P2P を実装しない。まずはローカル対戦でゲームルールと UI を固める。

P2P を検討する段階では、WebRTC を第一候補にする。WebRTC では接続開始のために offer / answer / ICE candidate の交換が必要になるため、完全にサーバーなしにするか、最小限のシグナリングだけサーバーや API に任せるかを改めて決める。

当面の方針は次の通り。

- v1: 同一ブラウザ内のローカル対戦
- v2: ゲーム状態と操作イベントを通信しやすい形に整理
- v3: WebRTC P2P の技術検証
- v4: passkey ログインとオンライン対戦の統合

## 学習方針

このリポジトリでは、実装と学習を分けすぎない。

新しい技術を使うときは、次の進め方を基本にする。

1. 小さい範囲で実装する
2. その範囲だけ動作確認する
3. 何が分かったか docs に残す
4. 次の範囲に進む

特に学習対象として重視するもの。

- JavaScript / TypeScript のデータ構造と型設計
- React の状態管理とコンポーネント設計
- Vite によるフロントエンド開発
- Amplify Gen 2 の backend 定義
- Cognito custom auth
- WebAuthn / passkey
- 将来的な WebRTC P2P 通信

TypeScript は段階的に導入する。最初から全ファイルを TypeScript 化するよりも、ゲームロジックのように型の恩恵が大きい部分から導入する。

## 段階的ロードマップ

### Phase 1: ゲームルールの整理

- `docs/simpei rule.md` を実装時のルールの出発点にする
- 盤面、駒、手番、勝利条件はこの文書に従う
- 未確認の公式ルールが見つかった場合は、先に docs を更新してから実装を変える

### Phase 2: ローカル対戦の実装

- ゲームロジックを実装する
- React で盤面 UI を作る
- 2人が交互に操作できるようにする
- 勝敗判定とリセットを実装する
- ルールロジックは UI や認証から分離してテストする

### Phase 3: ゲーム状態の整理

- 盤面状態をシリアライズできる形にする
- 操作をイベントとして表現できるようにする
- 将来の P2P 対戦に備えて、UI とゲームロジックの依存を整理する

### Phase 4: passkey 認証の実装

- passkey 登録フローを作る
- passkey ログインフローを作る
- Cognito custom auth と接続する
- 認証済みユーザー情報をフロントで扱えるようにする

### Phase 5: P2P 対戦の検証

- WebRTC の接続検証を行う
- offer / answer / ICE candidate の交換方法を決める
- ゲーム操作イベントを P2P で同期する
- 切断や再接続時の扱いを検討する

## 未決定事項

現時点では、次の内容は未決定とする。

- 引き分けや投了の扱い
- オンライン対戦時のホスト権限
- P2P のシグナリング方法
- 対戦履歴を保存するかどうか
- ユーザープロフィールを持つかどうか
- Notes サンプルをいつ削除または置き換えるか

未決定事項は、実装直前に決める。ゲームルールの追加確認が必要になった場合は、コードを先に変えず、`docs/simpei rule.md` を更新してから実装する。
